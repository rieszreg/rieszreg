Tracer

The user writes the operator m(alpha)(z) with +, -, scalar *, and calls to alpha(...). rieszreg.estimands.tracer passes a Tracer in as alpha; each call returns a single-term LinearForm, and Python’s operator overloading composes them. The result has .as_pairs() returning [(coef, point), ...], which the default FiniteEvalEstimand.augment consumes to build the augmented dataset. Operations outside the linear-form algebra (squaring, integrals, constants) raise at trace time.

Adding an estimand

Estimands belong in rieszreg.estimands.base, not in the implementation packages. Built-ins subclass FiniteEvalEstimand and supply both an m operator and a vectorised augment(features, ys=None) override:

class MyEstimand(FiniteEvalEstimand):
    m_bar = 0.0  # closed-form E[m(alpha=1)(Z)]

        cov = tuple(covariates)
        self.param = param
        self.treatment = treatment
        self.covariates = cov

            def inner(z, y=None):
                x_kwargs = {k: z[k] for k in cov}
                return alpha(**{treatment: param, **x_kwargs})
            return inner

        super().__init__(
            feature_keys=(treatment, *cov), m=m, name=f"MyEstimand({param})",
            factory_spec={"factory": "MyEstimand", "args": {"param": param, "treatment": treatment, "covariates": list(cov)}},
        )

Subclasses that don’t override augment inherit the per-row Tracer-based default — fine for prototyping but slow. Then:

  1. Add to _FACTORY_REGISTRY in rieszreg/python/rieszreg/estimands/base.py so the round-trip serialization registry knows about it.
  2. Export from rieszreg/python/rieszreg/__init__.py.
  3. Add R wrappers in rieszboost/r/rieszboost/R/rieszboost.R, krrr/r/krrr/R/krrr.R, and forestriesz/r/forestriesz/R/forestriesz.R (R wrappers re-export from rieszreg via reticulate).
  4. Update docs/estimands.qmd with a row in the built-ins table and a runnable example in both languages.
  5. Add a test under rieszreg/python/tests/.
  6. Add a worked example under one of the implementation packages’ examples/.

Adding a Loss

Losses also belong in rieszreg, not the backends.

  1. Add the class to rieszreg/python/rieszreg/losses/<name>.py.
  2. Wire into rieszreg/python/rieszreg/losses/__init__.py (export + loss_from_spec registry).
  3. Add R wrappers.
  4. Update docs/losses.qmd with a runnable example.
  5. Test in rieszreg/python/tests/.
  6. Implementation packages whose backend supports the new loss should add a smoke test confirming it works on a small DGP.

Adding a backend

Pick the entry point that matches your learner’s natural loss decomposition. Both Protocols live in rieszreg/python/rieszreg/backends/base.py and both return a FitResult whose predictor implements predict_eta and predict_alpha.

Augmentation-style — Backend.fit_augmented(aug_train, aug_valid, loss, …). For learners that minimize a per-evaluation-point loss (kernel ridge, gradient boosting). The orchestrator hands you a precomputed AugmentedDataset of (a, b) coefficients at evaluation points; you don’t see the estimand. Reference implementations: XGBoostBackend, SklearnBackend (rieszboost), KernelRidgeBackend (krrr), AugForestRieszBackend (forestriesz).

Moment-style — MomentBackend.fit_rows(rows_train, rows_valid, estimand, loss, …). For learners that minimize a per-original-row loss (random forests, neural nets). You receive raw rows + the estimand and call rieszreg.trace(estimand, row) per row to compute moments. Reference implementation: ForestRieszBackend (forestriesz).

Implementations advertise at most one method; the orchestrator dispatches via hasattr and defaults to fit_augmented if both are present.

Register the predictor’s loader so save/load round-trip works:

from rieszreg.backends import register_predictor_loader
register_predictor_loader("my-kind", MyPredictor.load)

A new implementation package (e.g. rieszneural) follows the contract in rieszreg/DESIGN.md, Part B: depend on rieszreg, implement one of the two Protocols, satisfy the sklearn-conformance subset, contribute docs pages here.

Living-doc rule

Public-API changes update both README.md and the relevant docs/*.qmd page in the same commit. On bilingual pages, update both the Python and R tabs.

The pre-commit hook at .githooks/pre-commit (currently per-package; a meta-project version is on the roadmap) enforces this. Activate per clone:

git config core.hooksPath .githooks

The hook also greps the staged diff for two doc anti-patterns and blocks if it finds them: design-decision metacommentary (sentences describing the API’s negative space, e.g. what’s omitted, deferred, or rejected) and AI-flavored hedges (editorial framing, sweeping qualifiers, ornamental noun phrases). Match the ngboost user guide tone: direct, instructive, short.

For internal refactors that don’t touch any signature, test additions, or comment fixes: git commit --no-verify.
